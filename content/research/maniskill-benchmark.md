从2月末开始,我一直在折腾一件事 —— 在 ManiSkill 之上搭一个属于自己的 RL benchmark。听起来很学术,但具体做的事情其实很土:**让 Panda 机械臂在一张虚拟桌子上,把可乐放进盒子里、把笔放进笔筒、把眼镜盒推到一边**。

到今天,这个 benchmark 已经有 50+ 个任务,从单步的 `LiftApple-v1` 到长程多阶段的 `BreakfastTable-v1`。但比任务数量更让我意外的,是一路上踩过的那些坑。

## 写第一个任务,就花了一周

第一个任务是 `PickCola-v1` —— 让机器人抓起一罐可乐。听上去最多两百行代码,但真要写出来,会发现远不止:

- **`_load_scene`**:把可乐建出来,顺便给它一个合理的 collision mesh。primitive box 当然简单,但训练出的策略一看就是"在抓积木"。
- **`_initialize_episode`**:每个 episode 把可乐随机摆在桌子的某个区域,随机旋转。范围太小过拟合,太大根本学不会。
- **`evaluate`**:成功的判定要写成 GPU 友好的张量操作 —— 因为整个仿真是在并行的 1024 个环境里跑的,任何 `if` 都得变成 `mask`。
- **`compute_dense_reward`**:这个最折磨。后面专门讲。

第一周写完后,模型收敛了,但策略很丑 —— 机械臂会用一种"抽风"的姿势把可乐拍翻在桌上,刚好压住成功判定的那一帧。我才意识到,**所谓 reward shaping,是真的能把模型教坏的**。

## staged reward,以及为什么它必须分段

最初我写过一个"全合一"的稠密奖励:

$$
r = -\|p_{ee} - p_{obj}\| - \|p_{obj} - p_{goal}\| + \mathbb{1}[\text{success}]
$$

这玩意训出来的策略,会把物体推过去而不是抓过去 —— 因为推比抓容易,且 $\|p_{obj} - p_{goal}\|$ 一项不在乎物体是不是在夹爪里。

后来我老老实实把奖励切成阶段:

```python
def compute_dense_reward(self, obs, action, info):
    # Stage 1: 接近物体
    reach_reward = 1 - tanh(5 * dist_ee_to_obj)

    # Stage 2: 抓住物体后,接近目标
    is_grasped = self.agent.is_grasping(self.obj)
    place_reward = (1 - tanh(5 * dist_obj_to_goal)) * is_grasped

    # Stage 3: 物体到位且夹爪松开
    success_reward = info["success"] * 5.0

    return reach_reward + place_reward + success_reward
```

写到第三十个任务的时候,这个三段式的骨架几乎成了我的肌肉记忆。但它从来不是"套用就行" —— 长程任务里,得拆出第四段、第五段;**不同任务的 stage 之间不是叠加,而是接力**,前一段不达到阈值,后一段的奖励要乘 0,否则模型会去刷"轻松的那一段"。

这个教训我反复吃过。

## 真正让我兴奋的是:从一张照片开始

任务一个一个手写,到第十几个的时候开始觉得无聊。所有的 `_load_scene` 都长得很像,所有的 reward 都是那个三段式。**重复劳动一旦超过了某个阈值,就该写工具,而不是继续手写**。

于是有了 `scene_to_task` 这条 pipeline:

1. 给一张真实桌面的照片;
2. 用 VLM 识别桌上有哪些物体,把每个物体抠成单独的图;
3. 每张图丢给 image-to-3D 的 API,得到 GLB 文件 + 真实尺寸;
4. 根据"用 Panda 把 X 放到 Y 上"这种自然语言描述,自动生成一个 ManiSkill 任务文件。

听起来很顺,实际上每一步都有它自己的小恶魔:

- 第 2 步,VLM 经常会把一个杯子和它的影子识别成两个物体;
- 第 3 步生成出来的 mesh,convex hull 经常和视觉外形差很多 —— 一个圆柱形的笔筒最后碰撞体可能是一个矮胖的多面体;
- 第 4 步是最难的,因为"把笔放进笔筒"和"把可乐放进盒子"在自然语言层面只是宾语不同,但 reward 函数完全不一样 —— 一个看的是开口朝向,一个看的是 AABB 包含关系。

我现在的处理方式很笨:**用 LLM 生成任务草稿,再人工调一版**。这样平均一个新任务从两小时压到了二十分钟,够用。

## 一个让我反复 debug 的细节

有一个坑值得专门拎出来:**所有从外部 mesh 加载的物体,必须用 `add_convex_collision_from_file`,而不是 primitive 替代**。

我最初为了省事,把一些细长物体(笔、牙膏)用 capsule 近似,视觉用真实 mesh。结果训练时,**夹爪能"穿过"视觉外壳抓住中心的 capsule** —— 因为视觉和碰撞根本没对齐。

更阴险的是,这种 bug 在单环境调试时不容易看出来,只有在 1024 个并行环境跑了几小时之后,才会发现策略学会了一个"反物理直觉"的抓法。

后来我把这个写成了项目里的硬规则:

> 所有物体的 collision 用 `add_convex_collision_from_file`,visual 和 collision 共用同一个 `pose=rot`,静态道具的 z = 0.001,主物体在 `_after_reconfigure` 里做一次 z 校准。

这条规则现在像迷信一样地贴在我项目的 memory 里。但每次有新人(或者新版的 LLM 助手)犯同一个错误时,我就庆幸自己写下来了。

## 一个不算总结的总结

做这个 benchmark 的过程让我意识到 —— **scale 在工程里的真正含义,不是任务数量从 10 变成 50,而是把"造一个任务"的边际成本从两小时压到二十分钟**。

任务数量是结果,工具链才是因。

下一篇打算写写在这个 benchmark 上跑 PPO 的一些观察 —— 哪些任务好学,哪些任务训了一千万步还是抓不住可乐。提前剧透一句:**长程任务里 reward 设计的难度,是单步任务的几个数量级以上**。

> 写代码很容易让人产生一种错觉:每一行都是为了解决眼前的问题。但回头看,真正有价值的,几乎都是那些"为了让下一次更轻松"而写下的代码。
