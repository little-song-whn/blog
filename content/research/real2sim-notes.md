最近在做一个 Real2Sim 平台,把真机采集的数据"还原"成可以在仿真器里复现的场景。听上去就像把现实拍成视频再还原成 3D 场景 —— 实际上比这复杂得多。

## 表面上是 3D 重建,实际上是物理一致性

很多人以为 Real2Sim 等于"做一个看起来像的场景"。视觉上像就够了吗?完全不够。

举一个最简单的例子:**一个杯子的质量**。

- 真机里,机器人抓起这个杯子时,它的力反馈是 12N。
- 仿真器里,如果你给杯子一个错误的密度,机器人会在仿真里觉得它"飘"或者"太重",训练出来的策略迁移回真机就会失败。

所以 Real2Sim 的核心从来都不是 "**长得像**",而是 "**碰上去也像**"。

## 三个比想象中更难的东西

1. **接触动力学的还原**。摩擦系数、恢复系数、阻尼 —— 这些参数在真机里没法直接读出来,只能通过若干次"实验回放"反推。而反推的优化目标本身就很 ill-posed。

2. **传感器噪声的建模**。仿真里给的图像太干净了。如果不在仿真里加上和真机相似的高斯噪声、镜头畸变、运动模糊,策略学到的特征就是"完美光照下的特征"。

3. **时间的同步**。真机的控制频率、感知频率、动作执行延迟,都不是仿真里默认的那种"瞬间响应"。这一点最容易被忽视,但影响极大。

## 一个让我反复 debug 的场景

前阵子我在标定一个夹爪的 PD 控制器。仿真里我把刚度调到 800,机器人能稳稳抓起物体;但真机上同样的策略,夹爪一直在"颤抖"。

调了一下午才意识到 —— **真机里夹爪的电机有 50ms 的执行延迟**,而仿真里我用的是零延迟模型。

修法很简单:在仿真里加一个 action delay buffer。

```python
class DelayedActuator:
    def __init__(self, delay_steps=5):
        self.buf = collections.deque(maxlen=delay_steps)

    def step(self, action):
        self.buf.append(action)
        return self.buf[0] if len(self.buf) == self.buf.maxlen else None
```

但意识到这个问题花了我一整天。

## 一句不算总结的话

做 Real2Sim 越久,越觉得它和"做一个引擎"是两件事。引擎是工程,Real2Sim 是**调音师** —— 你不是在写代码,你是在让虚拟世界和真实世界唱同一首歌。

而它们音准对不上的地方,就是你下一篇 paper 的 motivation。
